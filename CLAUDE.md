# CLAUDE.md

이 파일은 이 저장소에서 작업하는 Claude를 위한 안내다.

## 프로젝트

웹에서 **대본 기반**으로 유튜브 편집을 최대한 편하게 하는 도구.
음성이 아니라 대본을 기준으로 여러 조각 영상을 잘라 본편을 조립하는데, 그 과정이
지나치게 수작업(노가다)이라 이를 자동화·간소화하는 것이 핵심 동기다.
큐시트(JSON)를 계약으로 삼아 편집 결과를 자동 렌더로 연결한다. 전 스택 TypeScript, pnpm 모노레포.

**북극성(사용자 선언, 2026-07-05): "내 전용 Vrew".** 원본 영상들을 던지면 (1) 초벌
편집본(큐시트)이 자동 생성되고, (2) 브라우저에서 비디오 에디터 수준의 UX로 다듬고,
(3) 바로 렌더까지 끝나는 개인 특화 에디터. 범용 Vrew와의 차별점 두 가지 — 대사 없는
영상에서 작동(시각 기반 초안 생성; Vrew의 전사 기반 편집은 이 콘텐츠에서 무용)하고,
사용자의 편집 문법(컷 리듬·샷 어휘·기승전결)이 내장된다. Vrew의 나머지(멀티트랙,
이펙트, 범용 템플릿)는 좇지 않는다 — "초안 자동화 + 터치업 에디터"가 전선이다.

**실사용 콘텐츠 예시(오해 방지용 — 반드시 읽을 것): 뜨개질 브이로그.** 원본 영상은
뜨개질하는 손/작업 모습을 찍은 것으로 **대사(내레이션)가 없다**. 사용자가 스크립트를
따로 써서 그걸 화면 자막으로 얹는 편집이다. 즉 "음성이 아니라 대본 기준"이란 말은
"오디오 속 발화 구간을 찾아 대본과 맞춘다"는 뜻이 **아니라**, "대본(자막) 한 줄이
묘사하는 동작·장면과 시각적으로 맞는 원본 영상 구간을 찾는다"는 뜻이다. 그래서:
- 이 프로젝트에서 "스크립트-영상 매칭"을 다룰 땐 **오디오 전사(transcript/Whisper 등)
  기반 매칭은 기본적으로 부적합**하다 — 원본에 대사가 없으므로 전사할 대상 자체가 없음.
  매칭은 **시각적 내용** 기준이어야 한다(예: Claude Vision으로 프레임을 보고 판단).
- 외부 도구(예: `claude-real-video`)를 이 프로젝트에 적용하는 걸 검토할 때, 그 도구가
  오디오/발화 중심 기능을 내세우더라도 **이 프로젝트엔 그 기능이 필요 없다**고 기본
  가정해라 — 씬(프레임) 추출처럼 시각 기반 기능만 골라 쓰는 방향으로 판단할 것.
- **대본 순서 = 영상 시간 순서다(브이로그 특성).** 매칭은 영상 전체를 뒤지는 전역
  검색/재배열 문제가 아니라 **순서 보존(단조) 정렬** 문제 — 앞에서부터 훑으며 경계
  지점만 찾으면 된다. 전역 매칭 로직을 만드는 건 과설계.
- **원본은 길이가 제각각(짧으면 3초, 길면 40분)이고 움직임이 작다.** 컷 전환 없는
  롱테이크에서 뜨개질하는 손의 미세한 변화가 대부분이라, 일반적인 씬 전환 감지가
  잘 안 걸릴 수 있는 소재임을 전제로 접근할 것. 자동화의 핵심 산출물은
  "이 영상에서 쓸만한 순간이 몇 초 지점"이라는 **하이라이트 위치 뽑기**다.
- **얼굴 공개 정책(절대 규칙, 2026-07-07 사용자 확정): 얼굴은 턱끝까지만 노출한다** —
  "거의 턱끝이 나온다고 생각하면 될 듯" 기준. 입술(아랫입술 포함) 위가 보이면 위반이므로
  초안에 넣지 말거나(같은 클립의 안전 구간으로 대체) 세로 크롭으로 턱끝 라인까지 잘라야
  한다. 스캔(비전) 단계에서 얼굴 노출 여부를 반드시 판정·기록하고, 자동 크롭 제안이 기본
  동작. **가족 등 타인 얼굴도 동일 기준(턱끝까지만) 적용.**
- **10분 이상 긴 클립은 보통 둘 중 하나다**: (1) 빨리감기(배속)용으로 찍은 것 —
  schema의 `segment.speed`가 이미 표현 가능, (2) 하이라이트 구간만 골라 쓸 것.
  긴 클립을 만나면 "통째로 배속할지 / 어느 지점을 하이라이트로 쓸지"를 판단·제안해
  주는 것이 사용자를 가장 편하게 만드는 지점.

작업은 **완전 자율**로 진행한다(질문으로 막지 말 것) — 자세한 규칙은 아래 골 커맨드 섹션.

**현황 보고 규약**: `docs/STATUS.md`가 살아있는 현황 문서다(컴포넌트 지도·핵심 결정·진행
중·검증 수치). 마일스톤마다 STATUS.md를 갱신하고 git 커밋해라 — "지금 뭐가 어디까지
왔는지"를 묻는 질문엔 이 문서 기준으로 답한다. 상세 결정 근거는 docs/ 하위 문서에 남긴다.

## 아키텍처

### 패키지

```
packages/
  schema/   큐시트 타입 정의 + 런타임 검증 (single source of truth)  [구현됨]
  draft/    원본 폴더 -> 초벌 큐시트 자동 생성 (스캔+프레임 추출/조립, 비전 판단은 Claude) [구현됨]
  web/      큐시트를 편집/미리보기하는 웹앱 (큐시트 생산)             [구현됨]
  render/   큐시트를 받아 실제 영상 렌더링, ffmpeg 등 (큐시트 소비)   [구현됨]
```

### 데이터 흐름 (누가 만들고 누가 쓰나)

```
  사용자 Claude Code ──(MCP/소켓)──┐
                                   ├─▶  큐시트(JSON)  ──▶  render  ──▶  최종 영상.mp4
  사람 손편집(웹) ─────────────────┘        ↑ 웹 미리보기
```

큐시트가 편집의 **공유 상태(single source of truth)**다. 자연어 명령("음성 30%로 낮춰줘")과
웹에서의 손편집이 둘 다 같은 큐시트를 바꾸므로 충돌 없이 섞인다. render는 그 큐시트를 본편으로 뽑는다.

**자연어 명령은 사용자의 Claude Code가 처리한다 — 앱에 Claude API를 심지 않는다(추가 비용 없음).**
편집 브리지(MCP 서버 우선, 또는 로컬 소켓/HTTP)가 큐시트 조작 툴을 노출하고, 사용자가 자기
Claude Code에서 명령하면 Claude Code가 그 툴로 큐시트를 편집한다. 웹은 결과를 미리보기·손편집하는
뷰어/에디터. 스키마 표현력이 편집력의 상한이므로, 다룰 편집 종류가 늘면 schema도 함께 확장한다.

### 의존 방향 (누가 누구를 import 하나)

```
  web  ──▶  schema  ◀──  render
```

- `schema`가 계약(contract)의 중심. web·render 둘 다 `@cuesheet/schema`를 import한다.
- `schema`는 web/render에 **의존하지 않는다** → 순환 없음.
- 웹은 저장 전, 렌더러는 렌더 직전에 각각 `validateCueSheet`로 검증한다.
  같은 스키마라 웹이 통과시킨 큐시트는 렌더러도 반드시 통과한다.

## 핵심 규약 (반드시 지킬 것)

- **문서 목적 체계 (2026-07-08 사용자 확정):**
  - `docs/` = 외부 공개 대상이거나 리서치 리포트 성격의 문서 (예: 벤치마크/OSS 지형 리서치,
    아키텍처). 단 USER-GUIDE처럼 사용자 개인용은 예외적으로 한국어 유지.
  - **GitHub 위키** = 실험 정리 리포트 — "이 프로젝트가 왜 생겼고 어떤 실험을 거쳤나"가
    궁금한 사람이 읽는 곳 (편집 문법 역산, 씬 감지 실측·배제, 초벌본 v1~v4 실험 등).
  - CLAUDE.md/STATUS.md = 작업용 내부 문서, 한국어.
- **외부 공개 문서는 전부 영어로 쓴다.** GitHub 위키(기존 한국어 페이지들도 영어로 전환
  예정), 이슈/PR, 공개 README 등 저장소 밖으로 나가는 것 전부.
- **GitHub 위키 작성 시 본문에 제목을 반복하지 않는다** — 위키는 페이지명이 곧 제목으로
  별도 표시되므로, 본문 첫 줄에 같은 제목(H1)을 넣으면 중복으로 보인다. 본문은 바로
  내용부터 시작.


- **시간 단위는 초(second)**. 프레임 아님. 프레임 환산은 render가 `project.fps`로만 처리.
- **클립 경로는 파일명만** (`segment.clip`), 폴더는 `clipDir`로 분리. 폴더를 옮겨도 안 깨지게.
- **타입은 zod 스키마에서 파생**(`z.infer`). 타입을 손으로 중복 정의하지 말 것 → 드리프트 방지.
- 검증 실패 시 `필드경로: 이유` 형식의 메시지를 준다 (예: `segments[0].in: in < out`).
- **이모지 금지** (코드·주석·커밋·자막 텍스트 예시 전부).

## 명령어

루트에서:

```bash
pnpm install
pnpm -r build      # 전체 빌드
pnpm -r typecheck  # 타입 체크
pnpm -r test       # 테스트 (vitest)
```

특정 패키지만:

```bash
pnpm --filter @cuesheet/schema test
```

## 스택

- TypeScript 5.x (strict, `noUncheckedIndexedAccess`)
- 검증: zod 4.x
- 테스트: vitest
- 패키지 매니저: pnpm workspaces (`pnpm-workspace.yaml`)
- ESM (`"type": "module"`), 내부 import는 `.js` 확장자 명시

## 새 패키지 추가 시

- `packages/<name>/package.json`에 `"@cuesheet/schema": "workspace:*"`로 물린다.
- `tsconfig.base.json`을 extends 한다.
- schema를 거스르는 별도 타입/검증을 만들지 말 것. 규칙 변경은 schema에서 시작한다.

## 골 커맨드 오케스트레이션 (개발 워크플로우)

목표를 던지면 끝까지 완주하는 `/goal` 커스텀 커맨드를 둔다. 모델 역할 분리:

```
  두뇌 (Opus 기본)  ──지시──▶  실행자
  /goal 커맨드                builder(Sonnet): 구현
  계획·분해·통합·검증          scout(Haiku): 탐색·기계적 작업
  (설계 결정은 Fable 에스컬레이션)
```

- `.claude/commands/goal.md` — 두뇌. 기본 `model: claude-opus-4-8`. 계획·분해·위임·검증·보고만.
  **티어링 규약(2026-07-06)**: 일상 오케스트레이션은 Opus, 방향이 갈리는 설계 결정·상충
  리서치 조합·전제 오류 감지는 Fable로 에스컬레이션. 토큰 대부분은 서브에이전트가 쓰므로
  두뇌 모델은 판단 품질 기준으로만 선택.
- `.claude/agents/builder.md` — 구현 실행자(Sonnet).
- `.claude/agents/scout.md` — 탐색 실행자(Haiku).
- 스펙은 `docs/goals/<슬러그>.md`에 "한 판 정리"로 남긴다.

사용: `/goal <달성하고 싶은 목표>` → 요구사항 정리 → 스펙 → 구현 → 검증까지 자동.

## 편집 브리지 (Claude Code 연결)

`@cuesheet/bridge` = Claude Code가 붙는 MCP 서버. 사용자가 자기 Claude Code에서 자연어로
명령하면 이 툴들로 큐시트를 편집한다(추가 API 비용 없음). schema로 검증되어 항상 유효한 큐시트만 저장.

- 툴: `get_cuesheet`(현재 값 읽기), `update_cuesheet`(전체 교체+검증) — 후자가 "자유도"의 핵심.
  어떤 편집이든 새 큐시트를 통째로 계산해 넘기면 검증 후 반영.
- 등록: 루트 `.mcp.json`에 `cuesheet-bridge` 서버. 편집 대상은 `CUESHEET_PATH`(기본 `project.cuesheet.json`).
- 웹앱은 이 파일을 감시해 미리보기 갱신(예정).
- 승인 프롬프트 없이 흐르도록 `.claude/settings.json`에 브리지 툴·Edit/Write·주요 pnpm 명령을 allow.
  (자율 편집 싸이클용 — 안전 경계는 여전히 되돌리기 어렵거나 외부로 나가는 작업.)

## 현재 상태

- `@cuesheet/schema`: 타입 + `validateCueSheet` + 예제. segment에 오디오 `volume` 추가. 테스트 7개 통과.
- `@cuesheet/draft`: 지금까지 세션 워크플로우 스크립트로만 존재하던 초벌 편집 파이프라인을
  정식 패키지로 승격. CLI(`cuesheet-draft`) 2단계 — `scan <원본폴더> --out <작업폴더>`(iCloud
  미다운로드는 `blocks===0` 확인 후 건너뛰고, 로컬 실물만 ffprobe로 길이 수집 후 길이별 간격
  (15초 미만 2s/60초 미만 5s/300초 미만 15s/그 이상 60s)으로 시크 기반 640px 프레임 추출 →
  `manifest.json`), `assemble --manifest --moments --clip-dir --project-name --out`(Claude가
  프레임을 보고 작성한 `moments.json`을 zod로 검증 후 조립 규칙 적용 — quality>=3 채택,
  같은 클립 내 monotonousRange 있으면 30~60초 슬라이스를 speed 14 배속 커넥터로 삽입(에피소드당
  8개 상한), 클립 파일명순→in 오름차순 정렬 → `validateCueSheet`로 검증 후 저장, 실패 시
  `필드경로: 이유` 출력하고 exit 1). 라이브러리 export `scanFolder`/`assembleDraft`(순수 로직).
  테스트 7개 통과(quality 필터링/정렬/배속 커넥터 출력길이 계산/8개 상한/검증 실패 케이스 등).
  실제 닷믹스 원본 폴더로 scan 실행 검증 완료(로컬 18개/미다운로드 33개, 프레임 172장 추출,
  920초 롱테이크도 시크 기반이라 수 초 내 완료), assemble도 손으로 만든 moments.json으로
  E2E 검증(정상 조립·검증 실패 두 경로 모두 확인).
- `@cuesheet/bridge`: MCP 서버(get/update). 스모크 테스트로 Claude Code 연결·큐시트 읽기 확인. 테스트 4개 통과.
- `@cuesheet/web`: 손편집 에디터(뷰어에서 승격 완료). 프로젝트 설정·자막 스타일·세그먼트
  편집(in/out/speed/volume/subtitle, 추가/삭제/순서 이동)·BGM 편집. Vite 미들웨어로
